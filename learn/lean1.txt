String[] args

When you run

java Main hello world 123

The JVM calls:

main(new String[]{"hello", "world", "123"});

So inside main:

args[0] = "hello"
args[1] = "world"
args[2] = "123"

That's all it is. The JVM passes command-line arguments as a String[].

Yes. There are a few types of variables. Don't memorize them yet—just understand them.

1. Instance Variable (Field)

Belongs to each object.

class Car {
    String brand;   // Instance variable
}
Car c1 = new Car();
Car c2 = new Car();

c1.brand = "BMW";
c2.brand = "Audi";

Each object has its own copy of brand.

2. Local Variable

Declared inside a method or constructor.

Car(String brand) {   // <-- brand is a local variable (parameter)
    this.brand = brand;
}

or

void start() {
    int speed = 100;   // Local variable
}

Local variables exist only while that method/constructor is executing.

3. Static Variable (Class Variable)

Shared by all objects.

class Car {
    static int count = 0;
}

There's only one count, no matter how many Car objects you create.

We'll cover static in more detail later.

Now this.brand = brand
class Car {
    String brand;          // Instance variable

    Car(String brand) {    // Local variable (parameter)
        this.brand = brand;
    }
}
this.brand → the object's field.
brand → the constructor parameter.

So you're copying the parameter into the object's field.

Can a constructor and a method have the same name?

Yes.

class User {
    User() {}          // Constructor
    void User() {}     // Method
}

This is valid Java, although it's considered bad practice because it's confusing.

The compiler distinguishes them because:

Constructor → no return type
Method → has a return type (void here)

You just understood one of the most important concepts in Java.

class Animal {
    void speak() {
        System.out.println("Animal sound");
    }
}

class Dog extends Animal {
    @Override
    void speak() {
        System.out.println("Bark");
    }

    void wiggleTail() {}
}

Animal a = new Dog();

Now:

a.speak();

✅ Compiles because Animal has speak().

At runtime it prints:

Bark

because the actual object is a Dog.

a.wiggleTail();

❌ Compile error.

Why?

The compiler checks:

"Does Animal have a method called wiggleTail()?"

No.

So it refuses to compile.

Mental model

The reference type decides:

"What methods am I allowed to call?"

The object type decides:

"Which implementation of that method actually runs?"

This one sentence is worth remembering.

What does @Override do?

It tells the compiler:

"I intend to override a parent method."

If you accidentally make a mistake:

class Dog extends Animal {
    @Override
    void speek() { }   // typo
}

The compiler immediately errors because there is no speek() in Animal to override.

Without @Override, this typo would silently create a new method—harder to catch.

B. Runtime error

C. Prints Wagging
