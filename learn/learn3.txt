ArrayList

      30 → right

You create a new node:

Change pointers:

Before:

Only pointer changes.

Example:

Total: O(n)

Whereas if you already have the node reference:

❌ Not exactly.

A HashMap does not store the key as a pointer directly to the value.

The magic is hashing.

Imagine:

Internally something like this happens:

Step 1: Hash the key

Java takes:

"aadarsh@gmail.com"

and calculates a number:

hashCode()
        ↓
123456
Step 2: Find a bucket

HashMap has an internal array:

Bucket array

0
1
2
3
4
5
6
7
...

It calculates:

123456 % array_size

Suppose:

bucket = 5

Now it stores:

Bucket[5]

Key: "aadarsh@gmail.com"
Value: User object

So lookup:

users.get("aadarsh@gmail.com")

does:

key
 |
hash
 |
bucket index
 |
directly jump there
 |
get value

That's why average:

O(1)

Where your linked list idea comes in:

Inside a bucket, collisions can happen:

Example:

Bucket[5]

(key1,value1)
   |
(key2,value2)
   |
(key3,value3)

Historically this was a LinkedList.

Modern Java (Java 8+) can convert a long collision chain into a balanced tree.

So mental model:

HashMap

Array
 |
 |-- Bucket 0
 |-- Bucket 1
 |-- Bucket 2
       |
       └── key-value entries

Suppose:

Assume both go to bucket 3:

Bucket[3]

Java cannot overwrite because keys are different.

So it stores multiple entries there:

Bucket[3]

[A,100] → [B,200]

Historically:

Bucket
  |
  Node
  |
  Node
  |
  Node

This was a linked list.

Now lookup:

map.get("B")

Java does:

1. Hash the key
"B"
 ↓
hashCode()
 ↓
bucket 3
2. Go to bucket 3

It finds:

[A,100]
[B,200]
3. Compare keys

It checks:

"A".equals("B")

false

200

If too many collisions happen in one bucket:

Old:

A → B → C → D → E

Linked list lookup:

O(n)

Java converts it into a balanced tree:

       C
     /   \
    A     E

Lookup becomes:

O(log n)

This is why backend developers override equals/hashCode

Example:

class User {

    String email;

    @Override
    public boolean equals(Object obj) {
        User other = (User)obj;
        return this.email.equals(other.email);
    }

    @Override
    public int hashCode() {
        return email.hashCode();
    }
}

Now:

u1.email = "a@test.com";
u2.email = "a@test.com";

will behave like the same key.

Very important interview rule:

If you override equals(), you must override hashCode().

Because HashMap does:

hashCode() → find bucket
equals() → find exact key

If they disagree, HashMap breaks.

Short version:

By default, Java compares objects by reference.

User u1 = new User("a@test.com");
User u2 = new User("a@test.com");

u1.equals(u2); // false by default

Even though data is same, they are different objects.

In backend, we usually care about business equality.

Example:

User 1:
id = 101
email = a@test.com

User 2:
id = 101
email = a@test.com

Logically, they are the same user.

So we override:

equals()

to define:

"When should two objects be considered the same?"

and:

hashCode()

to define:

"Where should this object go inside HashMap/HashSet?"

Example:

Without override:

Set<User> users = new HashSet<>();

users.add(u1);
users.add(u2);

You get:

2 users

because Java thinks they are different objects.

With equals() + hashCode():

1 user

because Java understands they are logically the same.

Backend example:

Avoid duplicate users in a Set
Use User as a HashMap key
Compare database entities
